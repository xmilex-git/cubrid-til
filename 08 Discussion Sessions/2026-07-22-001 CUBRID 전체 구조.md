# CUBRID 전체 구조

## 시작 의견

사용자는 첫 BFS 후보 중 1번인 **CUBRID 전체 구조**를 선택했다. `1번...?`이라는 표현을 고려해, 아직 범위가 선명하지 않은 잠정 선택으로 이해하고 큰 그림부터 구체화한다.

## 질문과 합의

### Q1. 세 개의 BFS 후보 중 어떤 주제로 시작할까?

- 사용자 의견: `1번...?`
- 권장안: **CUBRID 전체 구조**. 이후 engine·개발·운영 주제를 이해하는 공통 지도가 된다.
- 합의: CUBRID 전체 구조를 현재 세션 주제로 선택한다.
- 근거: 깊이 0의 선수 지식 없는 engine frontier이며 다른 세부 코드 경로의 공통 맥락을 제공한다.

### Q2. 전체 구조를 어떤 요청 흐름으로 처음 연결해서 볼까?

- 사용자 의견: 동의했다. CUBRID 사람들은 client를 CAS, database server를 server라고 부르며 이 구조를 보통 3-tier 구조라고 설명한다고 덧붙였다.
- 권장안: 응용프로그램이 SQL 한 건을 보내고 결과를 받는 흐름을 기준으로 본다. 구성요소 이름을 실제 역할과 연결하기 쉽기 때문이다.
- 합의: SQL 한 건의 흐름을 기준으로 3-tier 구조를 본다. 단, `client`는 application 관점과 engine 관점에서 뜻이 달라 이를 구분한다.
- 근거: 공식 manual과 source에서 `cub_cas`는 middleware worker인 동시에 `cub_server`에 연결하는 database client process로 확인된다.

### Q3. 현업에서 “CAS”는 어느 process 범위까지 가리키나?

- 사용자 의견: CAS는 CAS이고 broker는 broker이며 JDBC driver 역시 별도 영역이다.
- 권장안: canonical term으로는 `cub_cas` process만 CAS라 하고, `cub_broker`까지 포함한 층은 broker middleware tier로 구분한다. 실제 process 경계와 장애 지점을 정확히 설명할 수 있기 때문이다.
- 합의: CAS, broker, JDBC driver를 서로 다른 구현 영역으로 엄격히 구분한다. 논리적 tier를 설명할 때 함께 배치되더라도 이름을 포괄적으로 사용하지 않는다.
- 근거: 공식 문서와 build target에서 `cub_broker`와 `cub_cas`는 역할과 executable이 분리되어 있다.

### Q4. 현업에서 3-tier의 세 tier는 정확히 무엇을 가리키나?

- 사용자 의견: CAS–broker–server를 3-tier라고 부른다.
- 권장안: application/driver, broker middleware(`cub_broker`와 `cub_cas`는 별도 process), database server의 세 논리 tier로 기록한다.
- 합의: 이 지식베이스에서 CUBRID 개발 현업 용어인 **CAS–broker–server 3-tier**를 사용하고 JDBC driver/application은 별도 영역으로 둔다. 공식 manual의 application–broker middleware–server 관점은 별도로 병기한다.
- 근거: 사용자의 현업 경험을 용어 관례로 기록하되, 공식 문서의 논리 구조 및 source의 process 경로와 구분한다.

### Q5. `CAS–broker–server`는 구성요소 열거인가, 요청 흐름 순서인가?

- 사용자 의견: 순서를 뜻하지 않으며, 실제 접속 흐름은 driver → broker → CAS → server가 맞다.
- 권장안: 현업의 tier 구성요소를 열거한 명칭으로 기록하고, 실제 application 접속 흐름은 driver → broker → CAS → server로 별도 표기한다.
- 합의: `CAS–broker–server 3-tier`는 영역 명칭으로 사용하고 실제 접속 흐름은 별도로 `driver → broker → CAS → server`라고 쓴다.
- 근거: source와 공식 문서에서 broker는 application 접속을 CAS로 중계하고 CAS는 server의 database client로 동작한다.

### Q6. 다음으로 CAS와 server의 SQL 처리 책임 경계를 볼까?

- 사용자 의견: CAS가 parse, semantic check, rewrite, optimizer, XASL compile을 담당하고 server는 XASL 실행만 담당한다고 설명했다.
- 권장안: 정상적인 `SELECT` 한 건에서 CAS가 맡는 compile 단계와 server가 맡는 execute 단계를 symbol 수준으로 추적한다. CAS를 database client라고 부르는 이유를 실제 작업 분담으로 연결할 수 있기 때문이다.
- 합의: 정상 `SELECT`의 책임 경계에서 CAS는 parse → semantic check → rewrite → optimizer → XASL generation → XASL to stream을 담당하고 server는 prepare/cache → stream to XASL → XASL execution을 담당한다. 단, server의 storage·transaction 등 실행 기반 역할을 부정하는 표현으로 확장하지 않는다.
- 근거: `db_open_buffer()`, `db_compile_statement()`, `do_prepare_select()`의 client path와 `sqmgr_execute_query()` → `qexec_execute_mainblock()`의 server path에서 확인했다.

### Q7. 현업 표현 `XASL compile`을 canonical term으로 어떻게 기록할까?

- 사용자 의견: 풀어 쓰는 데 동의했다. 또한 `compile`은 실제 현업 용어가 아니며, `XASL generation`처럼 단계별로 말하는 것이 현업 용어에 가깝다고 정정했다.
- 권장안: `XASL compile`을 사용하지 않고 실제 단계와 symbol에 맞춰 풀어 쓴다.
- 합의: `XASL compile`을 canonical term이나 현업 용어로 사용하지 않는다. 이후 사용자 정정에 따라 CAS의 `XASL generation → XASL to stream`, server의 `prepare → stream to XASL`을 우선 표현으로 사용한다.
- 근거: source에서 XASL tree 생성, byte stream 직렬화, server prepare/cache가 서로 다른 함수와 경계로 나뉜다.

### Q8. CUBRID 전체 구조 주제를 현재 범위에서 완료할까?

- 사용자 의견: 완료 여부 대신 용어를 추가 정정했다. `serialization → XASL to stream`, `prepare → stream to XASL`이 현업 용어에 더 가깝다고 설명했다.
- 권장안: 완료한다. 3-tier 용어, 실제 접속 흐름, CAS–server의 정상 `SELECT` 책임 경계, cache hit 경계 시나리오와 code path까지 확보했기 때문이다.
- 합의: 완료 여부는 아직 미정이다. 용어는 `XASL to stream`과 `stream to XASL`로 정정한다. 현재 source에서 prepare는 stream을 cache하고 실제 `stx_map_stream_to_xasl()` 호출은 execute path에 있다는 timing 차이도 병기한다.
- 근거: 현재 질문을 이해하는 데 필요한 큰 그림과 WORKFLOW의 8개 완료 조건을 충족했다. DDL/DML과 plan cache 내부 상세는 후속 깊이로 분리할 수 있다.

### Q9. 용어 정정을 반영한 현재 범위에서 주제를 완료할까?

- 사용자 의견: 완료에 동의했다.
- 권장안: 완료한다. 현업 용어와 실제 source timing의 차이까지 반영했고, 큰 그림의 정상·경계 시나리오와 code path가 갖춰졌다.
- 합의: CUBRID 전체 구조 주제를 현재 범위에서 완료한다. DDL/DML 및 plan cache 내부 상세는 후속 주제로 분리한다.
- 근거: DDL/DML 및 cache 내부 상세는 현재 depth 0의 전체 구조보다 깊은 후속 주제다.

## 정정 및 충돌

- `client = CAS`는 틀린 표현이 아니라 engine의 client/server 경계에서 쓰는 표현이다. 외부 application 관점에서는 application/driver도 client이므로 문맥을 명시한다.
- “server는 XASL 실행만 한다”는 SQL compile/execute 책임 경계의 요약이다. server의 storage·transaction·lock·log 책임까지 없다는 뜻으로 쓰지 않는다.
- `XASL compile`을 현업 용어로 본 초기 기록을 정정했다. 우선 용어는 `XASL generation`, `XASL to stream`, `stream to XASL`이다.
