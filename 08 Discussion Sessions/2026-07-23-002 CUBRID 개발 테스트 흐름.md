# CUBRID 개발 테스트 흐름

## 시작 의견

- 사용자는 이번 세션에서 CUBRID의 개발·테스트 흐름을 알아보려 한다.
- 이 주제는 development 트랙 깊이 0의 기존 BFS 후보이며, 이번에 선택되었다.
- `개발 테스트 흐름`은 로컬 unit test, 별도 testcase 저장소의 회귀 테스트, PR 정적 검사와 merge gate까지 포함할 수 있어 먼저 범위를 합의해야 한다.

## 질문과 합의

### Q1. 어떤 관점으로 개발·테스트 흐름을 추적할까?

- 사용자 의견: 먼저 현재 세션을 push한 뒤 계속 진행해 달라고 요청했다. 권장한 end-to-end 관점에 별도 이견 없이 진행 요청한 것으로 해석했다.
- 권장안: engine 코드 한 건을 수정한다고 가정하고 `로컬 빌드 → 영향 범위에 맞는 test 선택·작성 → 로컬 실행 → testcase PR → engine PR 검사·merge gate`의 end-to-end 흐름을 먼저 본다.
- 합의: 권장한 end-to-end 흐름으로 진행한다. 해석이 다르면 다음 답변에서 정정한다.
- 근거: source checkout에는 CMake로 build하는 `unit_tests/`의 module test executable과 별도 `cubrid-testcases`, `cubrid-testcases-private-ex` PR을 확인하는 GitHub workflow가 함께 존재하므로, 한 변경의 생명주기로 연결해야 각 테스트 층의 책임 경계를 설명할 수 있다.

### Q2. 테스트 층을 code 위치가 아니라 observable contract와 risk로 선택하는가?

- 사용자 의견: `unit_tests`는 거의 사장되었고 실제 테스트는 인증이 필요한 QA Home에서 확인한다고 정정했다.
- 권장안: 변경 파일의 directory만으로 고르지 않고, 내부 invariant는 module test executable, SQL 결과·오류·server/utility 동작은 외부 testcase로 검증하며 둘이 겹치면 둘 다 작성한다.
- 합의: 권장안을 철회한다. 현재 CUBRID 개발 테스트는 unit test pyramid보다 외부 testcase/CTP 기반 기능·비기능 검증을 중심으로 이해해야 한다. `unit_tests/`는 source에 남아 있는 사실과 실무에서 거의 사장되었다는 사용자 경험을 분리해 기록한다.
- 근거: QA Home에 인증해 최근 세 build를 자동 추출했을 때는 `sql`, `sql_debug`, `medium`, `medium_debug`, `shell`, `cci`, `cci_debug`를 확인했다. 이후 사용자 제공 화면에서 훨씬 넓은 category와 `unittest`, `unittest_debug`도 확인했으며 두 category는 각각 4건이었다. non-functional summary에는 YCSB, SysBench, TPC-C, TPC-W, basic performance, Dots stability가 있었다.

### Q3. 개발자가 testcase 변경을 준비하고 QA 자동화에 넘기는 지점은 어디인가?

- 사용자 의견: 전체 개발 lifecycle을 다음과 같이 설명했다. 외부 요청/개발자 판단으로 JIRA issue 작성 → `To-triage` 선별 → 담당자 할당 후 `Analysis` → 분석 종료 후 `Develop` → 개인 fork/topic branch 개발 → 대상 branch의 `release`, `optdebug` build에서 로컬 CTP SQL·medium → engine develop PR → `/run all`로 SQL·medium·shell → 자동 생성된 두 testcase PR에서 CI와 변경 범위 확인 → engine/testcase CI와 review → squash merge → QA Home daily regression → QA가 회귀 JIRA 등록 → 안정화된 변경이 정식 release에 포함된다.
- 권장안: engine 변경과 함께 재현·회귀 testcase를 준비하고, engine PR에서 자동 생성되는 `tc/pr-<PR 번호>` branch/PR에 testcase 변경을 반영한 뒤 CTP와 QA Home 결과를 확인하는 흐름으로 본다.
- 합의: 사용자가 설명한 흐름을 현재 canonical lifecycle로 채택한다. 다만 local release·optdebug SQL·medium은 사용자 현업 관례, PR bot/CI는 source와 PR 사례로 검증된 사실, release 선별·승인은 미확인 단계로 구분한다.
- 근거: `CONTRIBUTING.md`, GitHub/CircleCI workflow, PR #7516과 연결 testcase PR #3107·#3715, 사용자 지정 JIRA/CTP skill을 교차 확인했다.

### Q4. 기존 failure와 신규 regression은 어떤 열을 기준으로 판정하는가?

- 사용자 의견: `Fail Total`은 known failure를 포함한 전체 실패이고 `Fail New`가 기준 build 대비 새로 발생한 regression 후보라서 QA가 우선 확인한다는 해석이 맞다고 확인했다.
- 권장안: `Fail Total`은 알려진 failure까지 포함한 현재 실패 수, `Fail New`는 직전 기준 build 대비 새로 발생한 regression 후보로 보고, unstable·Verify Status와 함께 triage 여부를 결정한다고 해석한다.
- 합의: `Fail Total`은 known failure를 포함한 현재 전체 실패 수, `Fail New`는 기준 build 대비 신규 regression 후보이며 QA가 우선 triage한다. Unstable·Verify Status와의 결합 규칙은 다음 질문으로 분리한다.
- 근거: 사용자의 현업 경험과 QA Home의 total/new 분리 화면.

### Q5. Squash merge 뒤 JIRA issue는 언제 어떤 완료 상태로 전환하는가?

- 사용자 의견: squash merge하면 issue를 `Resolved`로 만들며 이때부터 QA 작업으로 넘어간다. QA가 issue 검증을 시작할 때 `Test`, 검증을 완료하면 `Closed`로 전환한다.
- 권장안: engine merge 완료와 정식 release 완료를 하나로 합치지 말고, JIRA의 code-complete 상태/resolution 전환과 release 포함 상태를 별도로 기록한다.
- 합의: `Develop → (squash merge) → Resolved → (QA 검증 시작) → Test → (QA 검증 완료) → Closed`로 정리한다. `Resolved`는 개발 완료와 QA ownership 시작, `Closed`는 QA 검증 완료의 경계이며 정식 release 포함과는 별개다.
- 근거: 사용자의 현업 경험. JIRA transition automation과 권한은 아직 확인하지 않았다.

### Q6. `Closed`의 QA 검증 범위에 daily regression까지 포함되는가?

- 사용자 의견: QA는 `Test` 과정에서 category에 맞는 TC를 등록하고, 그 TC는 이후 매일 지속적으로 daily regression에서 실행된다고 설명했다. 따라서 issue별 검증과 지속 회귀 안전망을 구분한다는 권장안은 어느 정도 맞다고 평가했다.
- 권장안: issue별 acceptance/repro 검증과 요구된 regression suite 통과를 `Closed` 기준으로 보고, 이후 지속되는 전체 daily regression은 `Closed` 뒤에도 별도로 동작하는 안전망으로 구분한다.
- 합의: `Test` 단계에서 QA가 category TC를 등록하며 issue를 검증하고, 이 TC는 `Closed` 뒤에도 daily regression corpus에 남아 매일 실행되는 지속 안전망으로 구분한다. “평생”은 일회성 TC가 아니라 지속 자산이라는 뜻으로 기록하되 비활성화 예외는 미확인으로 둔다.
- 근거: 사용자의 현업 경험과 QA Home의 category별 daily result 구조.

### Q7. `Closed` 전에 TC의 최초 성공 실행까지 확인하는가?

- 사용자 의견: QA가 TC의 실제 동작을 확인하며, 작동하는 TC만 testcase repository에 merge한다고 설명했다.
- 권장안: category TC 등록만으로 닫지 않고, 수정된 build에서 issue별 acceptance와 새 TC의 최초 성공 실행을 확인한 뒤 `Closed`로 전환한다고 본다.
- 합의: QA가 수정 build에서 TC의 동작을 확인하고, 검증된 TC만 testcase repository에 merge한다. 이 검증을 통과한 TC가 daily regression corpus에 편입된다.
- 근거: 사용자의 현업 경험.

### Q8. Engine merge 전 TC PR과 merge 후 QA TC는 같은 경로인가?

- 사용자 의견: engine PR 단계에서 처리되는 testcase PR은 engine 변경 때문에 기존 TC가 달라지는 경우만 다룬다. 해당 issue의 신규 TC는 QA 확인이 필요하므로 engine PR merge와 `Resolved` 이후 `Test` 단계에서 처리한다.
- 권장안: engine PR 단계에서 개발자가 준비하고 bot branch에 반영하는 필수 회귀 TC와, engine merge 후 QA가 `Test` 단계에서 보강·검증하는 추가 TC를 두 경로로 구분한다.
- 합의: 두 경로로 구분한다. Pre-merge bot 연계 testcase PR은 기존 TC 정합성 변경, post-merge QA `Test` 단계는 해당 issue를 위한 신규 regression TC의 작성·검증·merge를 담당한다.
- 근거: 사용자의 현업 경험과 TC merge gate의 순서 제약.

### Q9. Pre-merge 기존 TC 변경은 누가 승인하는가?

- 사용자 의견: 개발자가 기존 TC 변경을 제안하고, 담당자는 CODEOWNER로서 PR에 reviewer로 자동 추가되는 process라고 확인했다.
- 권장안: 개발자가 engine 변경으로 달라진 기존 TC와 expected result를 제안하되, testcase reviewer 또는 QA가 specification 변화인지 regression 은폐인지 확인한 뒤 승인한다.
- 합의: 개발자가 pre-merge 기존 TC 변경을 작성하고, 변경 path의 CODEOWNER가 자동 reviewer로 추가되어 검토·승인한다.
- 근거: 사용자의 현업 경험. 두 testcase 저장소의 `.github/CODEOWNERS`에서 공개 `/sql/`, `/medium/`, `/isolation/`과 private-ex `/shell/`, `/shell_heavy/` path rule을 확인했다.

### Q10. `Fail New`는 어떤 분류를 거쳐 regression issue가 되는가?

- 사용자 의견: 실제 engine regression은 JIRA issue, 불규칙한 TC 문제는 unstable case, 환경·인프라 문제는 test error로 분류하고 Verify Status에 결과를 반영한다는 설명이 맞다고 확인했다.
- 권장안: QA가 `Fail New`를 재현·분석해 실제 engine regression이면 JIRA issue를 만들고, 반복성이 낮은 TC 문제는 unstable case, 환경·인프라 문제는 test error로 분류한 뒤 Verify Status에 판정 결과를 남긴다고 본다.
- 합의: 권장한 분류 흐름을 현재 QA triage process로 채택한다. 세부 threshold와 Verify Status 값별 의미는 미확인으로 둔다.
- 근거: 사용자의 현업 경험과 QA Home의 unstable case, test error status, Verify Status 구분.

### Q11. 개발자 로컬 검증에서 shell을 보통 제외하는 이유는 무엇인가?

- 사용자 의견: shell 전체는 CI의 공용 인프라에서 병렬 실행하고, 실패한 case만 `cubrid-shell-run` workflow로 로컬 재현한다고 설명했다.
- 권장안: shell suite는 HA·utility·process·환경 의존 시나리오가 많고 실행 비용과 격리 요구가 커서, 개발자는 로컬에서 SQL·medium으로 빠른 사전 검증을 하고 shell 전체는 PR `/run all`의 전용 CI 환경에서 확인한다고 본다.
- 합의: shell 전체는 공용 CI가 병렬 실행하고, 개발자는 실패한 case만 로컬 focused runner로 재현·분석한다.
- 근거: 사용자의 현업 경험, CircleCI의 별도 shell executor·timing split, `cubrid-shell-run/SKILL.md`의 focused CTP 실행 절차.

### Q12. CI shell failure가 로컬에서 재현되지 않으면 어떻게 판정하는가?

- 사용자 의견: CircleCI REST API로 원 실행을 분석하거나 CircleCI의 failed-case rerun 기능으로 실패 case만 다시 실행한다. `develop`에서도 원래 실패하는 case일 가능성이 있으면 사내 Teams 게시판에 제보한다고 설명했다.
- 권장안: 동일 OptDebug revision과 testcase로 로컬 재현되면 engine/TC를 수정하고, 재현되지 않으면 build·revision·환경 차이를 먼저 제거한 뒤 반복 결과에 따라 unstable TC 또는 CI 환경 failure로 분류한다.
- 합의: 단일 local pass로 CI failure를 무효화하지 않는다. REST API 분석, failed-case rerun, develop baseline 비교를 거쳐 PR regression·unstable·CI 환경·기존 baseline failure를 분류하고, 공용 baseline 문제는 Teams에 제보한다.
- 근거: 사용자의 현업 경험. CTP의 active install 의존성과 CircleCI의 별도 OptDebug·공용 환경도 이 교차 확인의 필요성을 뒷받침한다.

### Q13. Develop baseline failure이면 현재 PR merge를 계속할 수 있는가?

- 사용자 의견: 일반적으로는 merge할 수 없다. 예외적으로 CTO 등 repository 관리자의 판정에 따라 현업에서 “강제 머지”라고 부르는 처리를 할 수 있으나, 보통 QA가 문제를 빠르게 고치는 동안 기다린다. 수정 후 각 저장소의 `develop`을 engine PR branch와 TC branch들에 merge하고 CI를 재실행한다. SQL·shell은 testcase `develop`이 아니라 해당 TC branch 기준으로 실행된다.
- 권장안: 동일 case가 동일 조건의 `develop`에서도 실패한다는 증거와 Teams 추적을 남기면 현재 PR이 만든 regression과 분리하되, merge 진행 여부는 code owner 또는 CI owner의 명시적 판단을 받는다.
- 합의: 일반 절차에서는 기존 baseline failure도 해결을 기다리고 최신 `develop`을 engine/TC branch에 merge한 뒤 재검증한다. 관리자 예외 판정에 따른 “강제 머지”는 드문 예외다.
- 근거: 사용자의 현업 경험. CircleCI의 `resolve-testcases`도 `tc/pr-<번호>` branch를 우선 선택하므로 testcase `develop` 수정은 TC branch로 merge해야 PR CI에 반영된다.

### Q14. 현업 용어 “강제 머지”는 GitHub 관리자 bypass merge인가?

- 사용자 의견: 맞다고 확인했다. Repository 관리자만 가능하며 일반 개발자는 수행할 수 없다.
- 권장안: Git의 force push와 구분해 `repository 관리자 승인에 의한 branch-protection 예외 merge`로 기록한다.
- 합의: 현업 용어 “강제 머지”의 canonical 설명은 `repository 관리자 승인에 의한 branch-protection 예외 merge`다. 관리자만 수행할 수 있고 일반 개발자는 실행할 수 없으며 force push와는 다르다.
- 근거: 사용자의 현업 경험.

### Q15. 현재 개발·테스트 흐름 세션을 마무리할까?

- 사용자 의견: 현재 topic을 완료하고 push해 달라고 요청했다.
- 권장안: 현재 topic을 완료 처리하고 동기화한다. JIRA intake부터 local CTP, PR/TC branch, CODEOWNER review, QA Test·daily regression, failure triage와 관리자 예외까지 중심 lifecycle이 합의되었다.
- 합의: 현재 topic과 session을 완료하고 validator·commit·pull --rebase·push 순서로 동기화한다.
- 근거: 남은 unknown은 suite별 상세 책임, TC retirement, 관리자 예외 merge의 사후 기록과 release 운영처럼 각각 별도 심화 topic으로 다루는 편이 canonical note 경계를 유지하기 쉽다.

## 정정 및 충돌

- `unit_tests/AGENTS.md`는 test binary가 CTest에 등록된다고 설명하지만, 기준 commit의 tracked CMake에서는 `enable_testing()`, `add_test()`, `catch_discover_tests()`가 발견되지 않았다. 현재는 “CMake가 test executable을 build한다”까지만 확정하고 공식 실행 방법은 미확인으로 둔다.
- 초기에는 module unit test와 외부 testcase를 동등한 테스트 층으로 설명했으나, 사용자가 `unit_tests`는 거의 사장되었다고 정정했다. QA Home의 현재 category도 외부 testcase 중심이어서 초기 설명을 철회했다.
- 자동 HTML 추출 결과만 보고 QA Home에 unit test category가 없다고 기록했으나, 사용자 제공 화면에는 `unittest`, `unittest_debug`가 각각 4건 존재했다. “없음”을 “존재하지만 전체 회귀 suite에서 비중이 매우 작음”으로 정정했다.
