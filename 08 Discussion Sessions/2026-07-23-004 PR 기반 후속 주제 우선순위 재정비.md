### Q2. PR 기반 우선순위를 BFS와 어떻게 함께 표현할 것인가

- 사용자 요청: 최근 PR을 기준으로 본인이 손댄 코드와 주제를 가장 높은 우선순위에 둔다.
- 적용안: `P0`은 최근 본인 구현과 직접 연결된 주제, `P1`은 그 구현의 선수·관찰 주제, `P2`는 기존 BFS 일반 확장, `P3`는 현재 코드 관심과 거리가 있는 process 세부로 분류한다.
- 경계: 우선순위와 BFS 승격 가능 여부를 분리한다. `P0`이어도 선수 지식과 깊이가 열리지 않았으면 `queued`로 유지한다.
- 결과: `Learning State.md` 큐에 우선순위 열을 추가하고, 기존 중복 항목은 구체화·병합했으며 새 주제는 선수 관계와 함께 추가했다.

## 관찰된 코드 근거

### Current develop snapshot

**기준 checkout:** `e1e81d600f604d0fc22ded3066186a1a9aaec184` (`develop`)

현재 checkout은 사용자 변경이 있고 `origin/develop`보다 10 commit 뒤처져 있어 pull하지 않았다. 따라서 merged PR #7487의 수정 결과와 open PR #7516·#7504의 최종 코드는 이 snapshot에서 검증할 수 없다. PR 설명과 head/merge commit은 후보 선정 근거이며, canonical 구현 사실은 각 주제를 다룰 때 최신 checkout으로 다시 검증한다.

**출처:** `src/query/query_executor.c:qexec_is_cached_scan_eligible`

```cpp
return specp->type == TARGET_CLASS
  && specp->access == ACCESS_METHOD_SEQUENTIAL
  && scan_op_type == S_SELECT
  && !mvcc_select_lock_needed && !fixed && !grouped;
```

PR #7441의 cached scan은 현재 snapshot에 존재한다. 이 symbol은 scan eligibility, MVCC lock, fixed/grouped scan과 page-copy read mode를 연결하므로 기존의 넓은 `Fixed grouped cached scan 정책`을 `Cached heap scan`과 `Grouped scan legacy` 두 주제로 분리할 근거가 된다.

**출처:** `src/query/query_executor.c:qexec_execute_mainblock_internal`

```cpp
if (subq->precomp_owner_regu == NULL)
  {
    continue;
  }
```

PR #7316의 `precomp_owner_regu` 기반 scalar subquery precompute가 현재 snapshot에 존재한다. 이전 세션의 `aptr_list` 학습과 직접 이어지므로 P0로 두었다.

**출처:** `src/query/memoize.cpp:memoize_get`, `src/query/query_executor.c:qexec_execute_nljoin_with_memoize`

```cpp
ret = xasl->memoize_storage->get ();
```

`memoize_storage`는 NL join executor 경로에 직접 연결되고 #6555·#6652·#6877에서 반복적으로 변경되었으므로 기존 `NL join memoization`의 언급 횟수와 우선순위를 높였다.

### Checkout과 최신 PR 사이의 차이

현재 `qo_apply_parallel_index_scan_threshold()`는 `ceil(sel * leaf_pages)`와 client-side 판정만 설명하며, `scan_try_promote_parallel_index_scan()`은 `spec->num_parallel_threads`를 신뢰한다. PR #7516은 여기에 histogram provenance와 server 실제 page 수 gate를 추가하는 open 변경이다. 따라서 이 주제는 `partially verified` 사실이 아니라 merge 전 설계를 포함한 검증 대기 후보로 기록했다.

현재 `pgbuf_wake_flush_waiters()`는 flush waiter를 dequeue한 뒤 `waiter_exists`를 정리하지 않는다. 이는 checkout이 PR #7487 merge보다 뒤처졌기 때문이다. 최신 동작을 설명할 때 이 snapshot을 canonical 근거로 사용하면 안 된다.
