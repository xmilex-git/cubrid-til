# SELECT SQL 실행 경로

## 시작 의견

사용자는 `SELECT` SQL이 들어와 실행되는 경로를 실제 CUBRID source에서 알아보자고 제안했다. 이전 세션에서 확인한 CAS의 parse·rewrite·optimizer·XASL generation과 server의 XASL execution이라는 책임 경계를 함수 호출과 네트워크 요청 단위로 확장한다.

## 질문과 합의

### Q1. 이번 코드 추적의 시작점과 끝점을 어디로 둘까?

- 사용자 의견: 권장 범위에 동의했다. 전체 경로를 `parser → semantic check → rewriter → optimizer → XASL generator → executor`라고 표현하곤 한다고 설명했다.
- 권장안: CAS가 driver의 execute 요청을 읽는 protocol handler부터 시작해 server가 결과 list를 만들고 CAS가 응답을 돌려주는 지점까지 추적한다. driver별 protocol 구현보다 engine의 SELECT 처리 경로에 집중하면서도 왕복 경계를 놓치지 않기 때문이다.
- 합의: CAS protocol handler부터 server의 결과 생성과 CAS 응답 반환까지 추적한다. 단계의 canonical pipeline은 `parser → semantic check → rewriter → optimizer → XASL generator → executor`로 사용하되, 실제 `prepare`와 `execute` 요청 경계를 함께 표시한다.
- 근거: 기존 노트는 `db_open_buffer()`부터 `qexec_execute_mainblock()`까지의 compile/execute 책임만 요약했고, CAS request dispatch와 결과 반환 경로는 아직 검증하지 않았다.

### Q2. 현업 표현의 `rewriter` 범위를 어디까지로 볼까?

- 사용자 의견: 현업에서는 `mq_translate()`를 rewriter라고 부른다.
- 권장안: `rewriter`를 `mq_translate()`가 담당하는 view expansion과 내부 `mq_rewrite()`를 포괄하는 논리 단계로 정의한다. source의 함수명 하나에만 고정하면 실제 compile 순서를 과도하게 축소하기 때문이다.
- 합의: 이 지식베이스에서 `rewriter`는 `mq_translate()` 단계의 현업 명칭으로 사용한다. `mq_rewrite()`는 rewriter 내부 함수로 구분한다.
- 근거: `db_compile_statement_local()`은 semantic check 뒤 `mq_translate()`를 호출하고, SELECT의 `mq_translate_helper()`는 view/virtual class 변환 후 `mq_rewrite()`를 호출한다.

### Q3. parser 내부를 어느 깊이까지 볼까?

- 사용자 의견: parser 내부로 더 들어가지 않고 다음 단계로 넘어가기로 했다.
- 권장안: 이번 end-to-end 추적에서는 SQL text가 `PT_SELECT` 중심의 `PT_NODE` tree로 바뀌는 경계까지만 보고 semantic check로 이동한다. lexer와 grammar production 내부는 parser 자체를 별도 주제로 볼 때 더 효율적이기 때문이다.
- 합의: parser는 `SQL text → PT_NODE` 경계까지만 다루고 semantic check로 이동한다.
- 근거: `ux_prepare()` → `db_open_buffer()` → `parser_parse_string_with_escapes()` → `parser_main()` → `yyparse()`가 parser 호출 경계이고, `select_stmt` grammar action이 `PT_SELECT` node를 만든다.

### Q4. 현업의 `semantic check`는 어느 범위를 뜻하나?

- 사용자 의견: `semantic check`는 `pt_semantic_check()` 전체 범위를 뜻한다. 다만 name resolution은 중요하기 때문에 때로 별도 단계처럼 구분해서 생각한다.
- 권장안: source의 `pt_semantic_check()` 경계에 맞춰 name resolution, WHERE/SELECT clause 규칙 검사, type check를 모두 포함한다. 이름·타입 검사만으로 좁히면 실제 `pt_check_with_info()`가 수행하는 검사를 누락하기 때문이다.
- 합의: canonical pipeline에서는 name resolution을 semantic check에 포함한다. 상세 설명과 디버깅 관점에서는 중요도를 반영해 `name resolution → 나머지 semantic validation/type check`로 하위 단계를 분리할 수 있다.
- 근거: SELECT는 `pt_resolve_names()` → `pt_check_where()` → `pt_semantic_check_local()`을 지나며, local SELECT check 안에서 GROUP BY·ORDER BY 등의 규칙과 `pt_semantic_type()`을 처리한다.

### Q5. rewriter를 어떤 SELECT 시나리오로 먼저 볼까?

- 사용자 의견: 권장안인 view SELECT를 선택했다.
- 권장안: view를 조회하는 SELECT를 먼저 본다. `mq_translate()`라는 이름과 역할이 가장 분명하게 드러나고, 이후 일반 base-table SELECT의 `mq_rewrite()`를 구분하기 쉽기 때문이다.
- 합의: `FROM view` SELECT의 `PT_SPEC`가 base-class query 또는 derived table로 바뀌는 경로를 먼저 추적한다.
- 근거: SELECT의 `mq_translate_helper()`는 CTE/path 변환 후 `mq_translate_local()` → `mq_translate_select()`으로 view/virtual class를 확장하고, 이후 내부 `mq_rewrite()`와 `pt_semantic_type()`을 호출한다.

### Q6. View query의 pushable/non-pushable 판단까지 상세히 볼까?

- 사용자 의견: 세부 판단 조건은 나중에 함께 볼 주제로 큐에 넣고 optimizer로 넘어가기로 했다.
- 권장안: 이번 end-to-end 흐름에서는 parent tree에 직접 merge하거나 derived table로 보존하는 분기까지만 이해하고 optimizer 단계로 이동한다. `mq_is_pushable_subquery()`의 세부 조건은 view rewrite 자체의 별도 심화 주제로 충분히 크기 때문이다.
- 합의: `View query pushability 판단`을 후속 주제 큐에 등록하고 현재 세션에서는 `qo_optimize_query()`로 이동한다.
- 근거: `mq_translate_tree()`는 `mq_fetch_subqueries()`로 view query를 얻고 `mq_substitute_subquery_in_statement()`에서 `mq_is_pushable_subquery()` 결과에 따라 직접 치환과 `mq_rewrite_vclass_spec_as_derived()`를 나눈다.

### Q7. Optimizer를 어떤 SELECT 시나리오로 볼까?

- 사용자 의견: 권장안인 두 table join과 선택 조건이 있는 SELECT를 선택했다.
- 권장안: 두 table join과 선택 조건이 있는 SELECT를 본다. query graph의 node·term·join edge, index 후보, join order라는 optimizer 핵심을 한 사례에서 연결할 수 있기 때문이다.
- 합의: 두 table join과 filter predicate가 있는 SELECT를 기준으로 `QO_ENV`와 `QO_PLAN`을 추적한다.
- 근거: `qo_optimize_helper()`는 rewritten `PT_SELECT`에서 query graph를 구성하고 join edge, equivalence class, index·partition 후보를 발견한 뒤 `qo_planner_search()`로 `QO_PLAN`을 선택한다.

### Q8. Optimizer의 cost/cardinality 계산까지 볼까, XASL generator로 넘어갈까?

- 사용자 의견: optimizer cost/cardinality와 join enumeration은 나중에 볼 주제로 큐에 등록하고 XASL generator로 넘어가기로 했다.
- 권장안: 현재는 `QO_NODE`·`QO_SEGMENT`·`QO_TERM` → `QO_PLAN` → `qo_to_xasl()` 경계까지만 잡고 XASL generator로 넘어간다. cost/cardinality와 join enumeration은 optimizer 자체의 별도 심화 주제로 충분히 크기 때문이다.
- 합의: `Optimizer cost/cardinality와 join enumeration`을 후속 주제 큐에 등록하고 현재 세션에서는 XASL generator를 추적한다.
- 근거: `pt_gen_optimized_plan()`이 `qo_to_xasl()`로 선택 plan을 XASL scan/access 구조에 반영하며, plan이 없거나 변환에 실패하면 허용되는 query에서 simple plan fallback을 사용한다.

### Q9. XASL proc/access 구조를 더 파고들까, XASL to stream으로 넘어갈까?

- 사용자 의견: NL join의 후행 table이 `scan_ptr`로 연결되는 `SCAN_PROC`로 생성된다는 점만 언급하고 넘어가자고 했다.
- 권장안: `BUILDLIST_PROC`, `outptr_list`, `ACCESS_SPEC_TYPE`, `scan_ptr`라는 실행 tree 경계까지만 잡고 XASL to stream으로 넘어간다. proc별 executor 동작은 server executor 단계에서 실제 소비 경로와 함께 보는 편이 연결이 좋기 때문이다.
- 합의: NL join의 inner/후행 table은 `SCAN_PROC` XASL로 만들어져 선행 쪽의 `scan_ptr` chain에 연결된다는 점을 기록하고 XASL to stream으로 이동한다.
- 근거: `pt_to_buildlist_proc()`가 root `BUILDLIST_PROC`를 만들고 `qo_to_xasl()`이 plan의 FROM/WHERE를 access spec과 scan/proc 구조로 변환한다.

### Q10. Server prepare는 정상 cache miss만 보고 execute로 넘어갈까?

- 사용자 의견: cache hit/recompile을 열지 않고 execute로 넘어가기로 했다.
- 권장안: 현재는 `XASL to stream → NET_SERVER_QM_QUERY_PREPARE → xcache_insert() → XASL_ID` 정상 cache-miss 경로만 보고 execute로 넘어간다. cache hit·recompile 생명주기는 이미 후속 주제 큐에 있으므로 현재 흐름을 유지할 수 있다.
- 합의: 정상 cache-miss server prepare에서 `XASL_ID`를 반환하는 데까지 보고 `CAS_FC_EXECUTE`와 executor로 이동한다.
- 근거: `xts_map_xasl_to_stream()`은 XASL tree를 linear stream으로 만들고, server의 `xqmgr_prepare_query()`는 정상 miss에서 `xcache_insert()` 후 `XASL_ID`를 반환한다. stream to XASL은 execute 시점이다.

### Q11. Executor 아래의 scan manager까지 내려갈까, 결과 반환으로 넘어갈까?

- 사용자 의견: scan manager 내부로 내려가지 않고 결과 반환으로 넘어가기로 했다.
- 권장안: 현재는 `qexec_open_scan()`과 outer `qexec_intprt_fnc`/inner `qexec_execute_scan`의 NL join 반복까지 보고 결과 반환으로 넘어간다. heap/index scan manager 내부는 access method별로 갈라지는 별도 심화 주제이기 때문이다.
- 합의: executor는 NL join 반복과 `QFILE_LIST_ID` 생성까지 보고 server reply, CAS fetch와 driver 응답으로 이동한다.
- 근거: `qexec_execute_mainblock_internal()`은 `scan_ptr` chain의 access spec을 열고 function vector로 root와 inner scan을 실행해 `QFILE_LIST_ID`를 만든다.

### Q12. 결과 반환 뒤 scan manager를 어떻게 다룰까?

- 사용자 의견: scan manager는 현재 흐름에서 더 내려가지 않고 후속 주제 큐에 등록하자고 했다.
- 권장안: `qexec_open_scan()` 아래의 heap/index scan open·next와 tuple 탐색은 access method별 분기가 크므로 별도 심화 주제로 둔다.
- 합의: `Scan manager와 access method 실행`을 engine 깊이 2 후속 주제로 등록한다.
- 근거: 현재 세션은 executor의 NL join 반복과 결과 반환 경계까지 도달했으며, scan manager 내부를 열지 않아도 SELECT end-to-end 책임 경계를 설명할 수 있다.

### Q13. 현재 SELECT 실행 경로를 완료할까?

- 사용자 의견: 완료 처리에 동의했다.
- 권장안: driver 요청부터 CAS compile, server execution, CAS fetch와 driver 응답까지 정상 경로가 연결됐으므로 현재 주제를 완료한다.
- 합의: `SELECT SQL 실행 경로` 세션과 [[CAS와 server의 SELECT 처리 경계]]를 `completed`로 변경한다.
- 근거: 완료 조건인 초심자 설명, 정상 시나리오, source commit·symbol·최소 excerpt, historical 분류, 사실·추론 구분, 미확인 사항, 관련 링크와 사용자 용어 정정을 모두 갖췄다.

## 정정 및 충돌

- “SQL이 들어간다”의 시작점은 driver, CAS socket/protocol handler, `db_open_buffer()` 중 하나일 수 있으므로 범위를 먼저 고정한다.
- pipeline의 단계명은 논리적 처리 순서이며, 각 단계가 하나의 함수이나 하나의 process에서 연속 실행된다고 미리 가정하지 않는다.
- `rewriter`를 source symbol `mq_rewrite()`와 일대일 대응시키지 않는다. 현업에서는 상위 단계인 `mq_translate()`를 rewriter라고 부른다.
- name resolution은 source상 semantic check 내부이지만 현업에서는 중요도 때문에 별도 단계처럼 말할 수 있다. 이는 책임 경계의 충돌이 아니라 설명 granularity의 차이다.

## 생성·갱신한 지식

- [[CAS와 server의 SELECT 처리 경계]]를 새 질문으로 `reopened`했다.
- [[CAS와 server의 SELECT 처리 경계]]에 CAS request dispatch, prepare/execute protocol 분리, server executor, 선택적 첫 fetch 경로를 추가했다.
- parser 단계의 입력 SQL text, `parser_main()`/`yyparse()`, 출력 `PT_NODE **` 경계를 추가했다.
- semantic check의 name resolution, WHERE/SELECT clause 검사, type check 경로와 실패 경계를 추가했다.
- rewriter의 CTE/path/view·virtual class/DBLink 구조 변환, 내부 `mq_rewrite()`, 타입 재검사 순서를 추가했다.
- view SELECT에서 cached view query를 가져와 pushable이면 parent tree에 merge하고, non-pushable이면 derived table로 보존하는 경로를 추가했다.
- optimizer가 XASL generation 내부에서 호출되어 `PT_SELECT → QO_PLAN → XASL_NODE`로 이어지는 구조와 optimizer-disabled fallback을 추가했다.
- 두 table join의 `QO_NODE`·`QO_SEGMENT`·`QO_TERM` mapping과 `qo_to_xasl()` 변환 경로를 추가했다.
- 일반 SELECT의 `BUILDLIST_PROC` root, `outptr_list`, `ACCESS_SPEC_TYPE`, `scan_ptr` 중심 XASL 구조를 추가했다.
- NL join inner table의 `SCAN_PROC`/`scan_ptr` 연결과 XASL to stream, server prepare/cache miss 경로를 추가했다.
- CAS execute, server-side XASL clone 준비, query executor 진입과 NL join function-vector 실행을 추가했다.
- Server가 packed `QFILE_LIST_ID`와 첫 list-file page를 반환하고, CAS client library가 이를 `DB_QUERY_RESULT` cursor로 감싸는 경로를 추가했다.
- `fetch_flag`에 따른 execute 응답의 첫 fetch와 별도 `CAS_FC_FETCH` → `fetch_result()` → `T_NET_BUF` → driver socket 반환 경로를 추가했다.

## 미해결 사항

- 현재 범위 안의 미해결 사항은 없다. Scan manager, fetch pagination과 cache/recompile은 후속 심화 범위로 남긴다.

## 나중에 다룰 주제

- View query pushability 판단 — `mq_is_pushable_subquery()`의 merge 가능 조건과 derived-table 보존 조건을 별도 검증한다.
- Optimizer cost와 join enumeration — cardinality/cost 추정, access path 비교, join 순서 탐색을 별도 검증한다.
- Scan manager와 access method 실행 — `qexec_open_scan()` 아래의 heap/index scan open·next와 tuple 탐색·predicate 적용을 별도 검증한다.
- Result fetch pagination과 overflow tuple — `CAS_FC_FETCH`의 추가 page 요청, cursor buffer와 overflow tuple 처리를 별도 검증한다.
